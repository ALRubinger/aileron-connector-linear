## What this repo is

`aileron-connector-linear` is an [Aileron](https://github.com/ALRubinger/aileron) connector for [Linear](https://linear.app). It is the first greenfield connector built on top of [`aileron-codegen`](https://github.com/ALRubinger/aileron-codegen); its presence (and overlay stability) is part of the codegen umbrella's acceptance criterion.

## Generated files are committed

`actions/<name>/action.md` is **generated by `aileron-codegen`** from `codegen/schema.graphql` + `codegen/gen.yaml`. The generated tree is **committed** so PRs are reviewable end-to-end.

Workflow:

1. Edit `codegen/schema.graphql` or `codegen/gen.yaml`.
2. Run `task generate`.
3. Review the resulting `actions/` diff like any other code change.
4. Commit both the overlay edit and the regenerated `actions/`.

The CI `generate:check` job runs `task generate` and `git diff --exit-code -- actions`. If the committed `actions/` drifts from what codegen would emit, CI fails. **Do not hand-edit `actions/<name>/action.md`** â€” your edit will not survive the next `task generate`. Put per-action prose somewhere the codegen prose emitter can consume (TBD) or override via `intent` / `action_name` / `execute_id` in `gen.yaml`.

## Codegen version pin

`Taskfile.yml`'s `generate` target invokes:

```
go run github.com/ALRubinger/aileron-codegen/cmd/aileron-codegen@latest ...
```

This is `@latest` while `aileron-codegen` is pre-release. The pin moves to a tagged semver once the overlay schema stabilizes (per [aileron#893](https://github.com/ALRubinger/aileron/issues/893) acceptance: two greenfield connectors shipped without overlay breaking changes â€” this connector is the first).

If you re-run `task generate` and the output drifts unexpectedly, check whether `aileron-codegen`'s `main` branch shifted behavior â€” `git log` on that repo will tell you.

## WASM target

The connector lives under `connector/` as its own Go module (`github.com/ALRubinger/aileron-connector-linear/connector`). `connector/main.go` builds for `GOOS=wasip1 GOARCH=wasm` and reads `{"op": "...", "args": {...}}` from stdin, writing `{"output": {...}}` or `{"error": {"class": "...", "message": "..."}}` to stdout. The runtime exec's the WASM, pipes stdin in, captures stdout.

The shipped dispatch is a **stub** â€” every known op returns `not_implemented`. The Linear GraphQL POST layer + per-op response shaping ships separately ([#915](https://github.com/ALRubinger/aileron/issues/915) follow-up PR).

When the dispatch handlers land, they will use Aileron's host imports for HTTP (the runtime injects the API key host-side at the network boundary â€” the connector binary never holds it; see ADR-0005). The Google connector ([`aileron-connector-google/connector/main.go`](https://github.com/ALRubinger/aileron-connector-google/blob/main/connector/main.go)) is the reference implementation.

## Op naming

The dispatch op names in `connector/main.go` must match the `op` field emitted into each `action.md`'s `[[execute]]` block. Codegen derives `op = snake_case(operationId)`, so:

| GraphQL field | `op` |
|---|---|
| `issue` | `issue` |
| `viewer` | `viewer` |
| `issueCreate` | `issue_create` |
| `commentCreate` | `comment_create` |

If you add an op, regenerate `actions/` first (`task generate`), check the emitted `op` value, then add the dispatch case.

## Workflow conventions

- **Branch + PR for everything except README.** Conventional commit titles (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`).
- **Squash-merge with `--admin --delete-branch`.** The repo is admin-merge-authorized.
- **Wait for CI before merging** â€” even with `--admin`, never merge while a check is pending.
- **No backwards-compat shims.** Pre-release.
- **No emojis in code or docs unless explicitly requested.**

## Releases

Pushing a `vX.Y.Z` tag triggers `.github/workflows/release.yml`, which uses SHA-pinned composite actions from [`ALRubinger/aileron-actions`](https://github.com/ALRubinger/aileron-actions) to build, sign, and publish the connector tarball plus per-action tarballs. The pipeline expects one repo secret:

- **`AILERON_SIGNING_KEY`** â€” base64-encoded ed25519 private key (PEM) for the publisher. Setup instructions in [`keys/README.md`](keys/README.md). The matching public key (`keys/publisher.pub`) is committed.

Before the first release, configure that secret in the repo's settings (Settings â†’ Secrets and variables â†’ Actions). Without it, the `sign-and-publish` step fails closed.

`task pack` runs the local equivalent of the build + tarball step (no signature unless `AILERON_SIGNING_KEY` is set in the environment). `task pack:hash` prints the SHA-256 of `connector.wasm || manifest.toml` â€” useful to verify what the release workflow will publish before pushing the tag.

The `0.0.0-dev` strings in `connector/manifest.toml` and every `actions/<name>/action.md` are **load-bearing placeholders** that the release workflow substitutes with the real version. The `sha256:bound-at-release` strings get substituted with the connector's content hash. Do not edit them by hand.

## ADRs

All ADRs live in [`ALRubinger/aileron/docs/adr/`](https://github.com/ALRubinger/aileron/tree/main/docs/src/content/docs/adr) and are referenced by number. The ones this connector touches most:

- **ADR-0002** â€” connector identity + content-addressed hash + `0.0.0-dev` placeholder substitution.
- **ADR-0005** â€” sandbox + credential mediation (the `Authorization` header is injected host-side; connector never sees the key).
- **ADR-0009** â€” per-call user approval gating (`[approval]` block in action.md, runtime asks user before dispatching).
- **ADR-0010** â€” idempotency declaration + runtime retry semantics.
